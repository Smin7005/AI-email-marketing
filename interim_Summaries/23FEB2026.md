# Implementation Plan: Nylas Email API Integration for Personal Email Senders

## Overview

Integrate Nylas Email API as a second email sending provider alongside the existing AWS SES integration. Users who only have a personal email address (Gmail, Outlook, etc.) can connect their account via Nylas OAuth. Users with custom domains continue using the existing AWS SES domain verification flow. The Add Sender dialog now has two tabs: "Personal Email" (Nylas) and "Custom Domain" (AWS SES).

**Current State:** Email sending uses AWS SES for verified domains, with Resend as fallback. The Senders page only supports SES domain verification.

**Key Decisions:**
- Personal Email (Nylas) uses OAuth popup for authentication (required by Gmail/Microsoft)
- Custom Domain (AWS SES) flow remains unchanged
- Send-emails function routes through SES for domain senders, Nylas for email senders, Resend as last fallback

---

## Architecture

```
User clicks "+ Add sender"
  |-- Personal Email tab
  |   |-- Enter email -> POST /api/senders (provider: nylas)
  |       |-- Open Nylas OAuth popup -> User authenticates
  |           |-- Callback: GET /api/nylas/callback -> Save grant ID -> Sender verified
  |-- Custom Domain tab (existing)
      |-- Enter email -> POST /api/senders (provider: ses)
          |-- Show DNS records -> User configures DNS -> Verify
```

## Potential Issues and Mitigations

### Issue 1: OAuth Requirement for Gmail/Microsoft
**Problem:** Nylas requires OAuth for Google/Microsoft accounts - users must authenticate in their browser.
**Mitigation:** Use popup window approach (`window.open()`) to keep the experience smooth. Poll for popup close to detect completion.

### Issue 2: Nylas Rate Limits
**Problem:** Rate limits depend on the underlying email provider (Gmail ~2/sec, Outlook ~4/sec).
**Mitigation:** Start with 200ms delay between sends. Can be adjusted per-provider if needed.

### Issue 3: Grant Expiration
**Problem:** Nylas grants can expire if the user revokes access or tokens expire.
**Mitigation:** `checkVerificationStatus()` calls `verifyGrant()` for Nylas senders and updates status to 'failed' if invalid.

### Issue 4: Nylas Pricing
**Problem:** Nylas charges per connected account (grant).
**Mitigation:** Monitor usage. Consider displaying a note about connected accounts in the UI.

### Issue 5: Popup Blockers
**Problem:** Browser popup blockers may prevent the OAuth window from opening.
**Mitigation:** The popup is triggered by a user click event (form submit), which most browsers allow. If blocked, show an error message with instructions.

## Follow-up 2:
Here's a step-by-step guide for setting up Nylas connectors (Not for free users):

Setting Up a Google Connector
  
  Part A: Google Cloud Console Setup
  1. Create a Google Cloud Project
    - Go to https://console.cloud.google.com/projectcreate
    - Enter a project name and create it
  2. Enable Required APIs
    - Go to APIs and services > Enable APIs and services
    - Search for and enable: Gmail API and People API
  3. Configure OAuth Consent Screen
    - Go to OAuth consent screen
    - Choose External user type > Create
    - Fill in app info, add nylas.com as an authorized domain
    - Add required scopes: userinfo.email, userinfo.profile, openid
    - Click Publish app to move out of testing mode
  4. Create OAuth Credentials
    - Go to Credentials > Create credentials > OAuth client ID
    - Application type: Web application
    - Add authorized redirect URI: https://api.us.nylas.com/v3/connect/callback
    - Click Create and save your Client ID and Client Secret
  5. Grant Nylas Access
    - Go to IAM & Admin > IAM > Add
    - Add support@nylas.com as an Owner

  Part B: Nylas Dashboard Setup

  6. Create the Connector in Nylas
    - Log in to https://dashboard.nylas.com
    - Go to Connectors in the left navigation
    - Click Create Connector > Choose Google
    - Paste your GCP Client ID and Client Secret
    - Configure scopes (at minimum: https://www.googleapis.com/auth/gmail.send for sending emails)
    - Save

  Part C: Set Up Your Callback

  7. In Nylas Dashboard, go to App Settings > Callback URI
    - Add your callback URL: https://yourdomain.com/api/nylas/callback
    - This is where our app's /api/nylas/callback route will receive the OAuth code

  Setting Up a Microsoft Connector

  The process is similar but uses Azure Portal instead of Google Cloud Console:
  - Register an app in Azure AD
  - Add redirect URI pointing to Nylas
  - Create the connector in Nylas Dashboard with your Azure Client ID and Secret

  Environment Variables After Setup

  Once done, you'll have everything needed:
  NYLAS_API_KEY=nyk_v0_xxx          # From Nylas Dashboard > App Settings
  NYLAS_CLIENT_ID=your-app-id       # From Nylas Dashboard > App Settings
  NYLAS_API_URI=https://api.us.nylas.com
  NYLAS_CALLBACK_URL=https://yourdomain.com/api/nylas/callback

## Follow-up 3:
Great question. Let me clarify how Nylas works:

Connectors are set up by YOU (the platform developer), not by your customers. You only need to create them once.

A Connector is the link between your Nylas application and an email provider (Google, Microsoft). It holds YOUR platform's OAuth credentials so that Nylas knows how to authenticate users against that provider.                                                                                                                                                                                                                                 
  Here's how it works in practice:

  You (one-time setup):
  1. Create a Google Cloud project with OAuth credentials
  2. Create a Microsoft Azure app with OAuth credentials
  3. Link those credentials to Nylas as "Connectors" in your Nylas Dashboard
  4. Done - never touch this again

  Your customers (simple experience):
  1. Click "Personal Email" tab in the Add Sender dialog
  2. Enter their email address (e.g. john@gmail.com)
  3. Click "Connect Email"
  4. A popup opens showing Google/Microsoft's standard "Sign in" page
  5. They sign in and click "Allow"
  6. Popup closes, their email is connected and verified

  Your customers never see Nylas, never configure connectors, never deal with APIs. They just see a familiar Google/Microsoft sign-in screen - the
   same experience they have when logging into any app with "Sign in with Google".

  Think of it like how Clerk works for your app: you set up Clerk once with Google/Microsoft OAuth credentials, and your users just click "Sign in
   with Google". Nylas connectors work the same way but for email sending permissions instead of login.

  So the complexity is entirely on your side during initial setup, and your customers get a very simple 2-click experience.
