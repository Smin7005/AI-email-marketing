# 1. Force Light Mode Across All Components

## Problem

Some components in the app become invisible when a user's OS or browser is set to dark mode. The root cause is that Tailwind CSS v4 applies `dark:` utility classes automatically when `prefers-color-scheme: dark` is detected from the OS. Most components (shadcn/ui cards, buttons, inputs) are hardcoded to light colors (bg-white, text-neutral-950, etc.) but the `dark:` variants in several files can override backgrounds and text in ways that create contrast problems. Browser-native elements (scrollbars, form controls) also flip to dark.

## Root Cause Analysis

- Tailwind CSS v4 uses the **media-query** dark mode strategy by default: `dark:` classes activate whenever the OS reports `prefers-color-scheme: dark`.
- `src/app/globals.css` defines no dark-mode CSS variable overrides, so `--background` and `--foreground` stay light, but Tailwind's utility dark variants still fire on individual elements.
- 3 files used `dark:` classes: `src/app/page.tsx`, `src/components/dashboard-header.tsx`, `src/components/campaigns/CampaignForm.tsx`.
- Browser-native elements (scrollbars, selects, date pickers) flip dark due to missing `color-scheme: light` declaration.

## Solution: Force Light Mode via globals.css

Switched Tailwind v4's dark mode from media-query strategy to class-based strategy by adding `@custom-variant dark` in globals.css. Since no `.dark` class is ever added to the `<html>` element, all `dark:` utilities are permanently inactive. Native browser elements are locked to light via `color-scheme: light`.

## Changes Made

### 1. src/app/globals.css (core fix)

Added after `@import "tailwindcss";`:

```css
/* Force light mode: switch dark variant to class-based so it never activates */
@custom-variant dark (&:where(.dark, .dark *));
```

Added `color-scheme: light;` inside `:root` block.

### 2. src/app/page.tsx (cleanup)

Removed all `dark:` variant classes (dead code):
- `dark:from-gray-900 dark:to-gray-800` from gradient background
- `dark:text-white` from headings
- `dark:text-gray-300` from body text
- `dark:bg-gray-800` from feature cards

### 3. src/components/dashboard-header.tsx (cleanup)

Removed `dark:bg-gray-800` from the header element.

### 4. src/components/campaigns/CampaignForm.tsx (cleanup)

Removed `dark:bg-gray-900` from the SelectContent dropdown component.

## Files NOT Changed

- `src/app/layout.tsx` — no `.dark` class is ever added to `<html>`, no change needed.
- All shadcn/ui components in `src/components/ui/` — already use hardcoded light colors.
- `src/components/dashboard/Sidebar.tsx` — intentionally uses dark slate-900 design; not a dark-mode issue.
- `src/app/(dashboard)/layout.tsx` — already hardcoded `bg-gray-50`.

## Verification

To verify the fix, switch your OS to dark mode and open the app. All pages should display in light mode (white/gray backgrounds, dark text). Native elements (scrollbars, dropdowns, form inputs) should also appear in light mode. The sidebar remains dark as an intentional design choice.

# 2. Auto-Refresh Collections Page After Collection Creation

## Problem

After a user selects leads on the leads page and creates a new collection via the "Save to Collection" modal, navigating to the collections page shows stale data — the new collection does not appear until the user manually presses F5 to refresh.

## Root Cause Analysis

- `src/app/(dashboard)/collections/page.tsx` is a Next.js server component with no `dynamic` or `revalidate` export. Next.js may serve a cached version of this route on navigation.
- The collection **detail** page (`src/app/(dashboard)/collections/[id]/page.tsx`) already has `export const revalidate = 0`, but the **list** page does not — leaving it subject to default caching behavior.
- `src/components/leads/AddToCollectionModal.tsx` `handleSave()` calls `onClose()` and `onSuccess?.()` on success but does not navigate the user to `/collections`. The user navigates there manually and hits a stale cached page.

## Solution

Two minimal changes consistent with existing codebase patterns (`router.refresh()` in CollectionCard, `revalidate = 0` in collection detail page):

### Change 1: src/app/(dashboard)/collections/page.tsx

Add at the top of the file:

```ts
export const dynamic = 'force-dynamic';
```

This forces Next.js to always run the server component fresh on every request, so `getUserCollections()` always returns up-to-date data from Supabase.

### Change 2: src/components/leads/AddToCollectionModal.tsx

1. Add import: `import { useRouter } from 'next/navigation';`
2. Add inside component: `const router = useRouter();`
3. In `handleSave()`, after `onSuccess?.();`, add: `router.push('/collections');`

This redirects the user to the collections page immediately after a successful save. Since Step 1 ensures `force-dynamic`, the page will always display the newly created collection.

## Files Changed

| File | Change |
|---|---|
| `src/app/(dashboard)/collections/page.tsx` | Add `export const dynamic = 'force-dynamic'` |
| `src/components/leads/AddToCollectionModal.tsx` | Import `useRouter`, call `router.push('/collections')` after success |

## Verification

1. Run `npm run dev`
2. Go to the leads page, select several rows, click "Save to Collection", create a new collection and save
3. Verify the browser automatically navigates to `/collections`
4. Confirm the new collection card appears immediately — no manual refresh needed

## Follow-up 1:
Here's what was changed:

- src/app/(dashboard)/collections/page.tsx — Added export const dynamic = 'force-dynamic'. Every navigation to /collections now always fetches fresh data from Supabase, identical to the pattern already used on the collection detail page.       

- src/components/leads/AddToCollectionModal.tsx — Added useRouter and called router.push('/collections') after a successful save. The user is now automatically redirected to the collections page where they'll immediately see their new collection.

## Follow-up 2:
Removed `router.push('/collections')` from `AddToCollectionModal.tsx`. Users stay on the leads page after saving so they can create additional collections without navigating back. The `export const dynamic = 'force-dynamic'` on the collections page is sufficient — the list will always show fresh data the next time the user navigates there.

# 3. AG Grid Column Separators and Resize Handle Visibility (Leads Page)

## Problem

The Leads page AG Grid table (ag-theme-quartz, v35) had no visible boundary lines between columns. Column resizing was already enabled (`resizable: true` in `defaultColDef`) but users couldn't discover this because the resize handle only appeared on hover and column dividers were invisible.

## Solution

Added CSS overrides to `src/app/globals.css` targeting `.ag-theme-quartz`:

- `--ag-header-column-separator-*` variables: draws a `1px` gray (#d1d5db) divider between every header cell at full column height.
- `--ag-header-column-resize-handle-*` variables: makes the blue (#93c5fd) drag handle permanently visible (40% height, 2px wide) at the right edge of each header instead of only appearing on hover.
- `.ag-theme-quartz .ag-cell { border-right: 1px solid #e5e7eb }`: extends the column dividers through the data rows so the full grid reads as a proper column grid.

## File Changed

| File | Change |
|---|---|
| `src/app/globals.css` | Added AG Grid column separator + resize handle CSS block |

## Follow-up 1:
Here's a summary of what was changed:

src/app/globals.css — Added a new CSS block for .ag-theme-quartz:

  - Gray 1px divider lines between every column header cell (--ag-header-column-separator-*)
  - The same divider lines extending through every data row (.ag-cell { border-right })
  - A permanent blue 2px resize handle at the header's right edge (--ag-header-column-resize-handle-*) — always visible, not just on hover, so users immediately know columns can be dragged to resize

  No JS changes were needed since resizable: true was already set in defaultColDef.

## Follow-up 1:

# 5. Enable Click-to-Select Rows on Leads and Collections Tables

## Problem

Row selection required users to click the small checkbox on the left of each row. Checkboxes are small and easy to miss, making multi-select tedious. Users want to click anywhere on a row to select or deselect it.

## Solution

Changed `enableClickSelection: false` → `enableClickSelection: true` in the `rowSelection` prop of `AgGridReact` in both table components. Checkboxes remain visible so users can still see which rows are selected at a glance.

## Files Changed

| File | Change |
|---|---|
| `src/components/leads/LeadsTable.tsx` | `enableClickSelection: false` → `true` |
| `src/components/collections/CollectionItemsTable.tsx` | `enableClickSelection: false` → `true` |

## Verification

Go to `/leads`, click anywhere on a data row (not the checkbox) — the row should be selected (checkbox ticked, row highlighted). Click again to deselect. Same behavior applies on the collection detail page.
