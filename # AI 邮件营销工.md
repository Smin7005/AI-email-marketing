## 0. 产品概览与范围

### 0.1 一句话概述

一款面向在澳大利亚做 ToB 生意的团队（尤其是本地线下服务商），基于内置澳大利亚商家数据库和外部 AI 大模型，自动为每个目标商家生成个性化开发邮件，并通过第三方邮件服务完成发送与效果统计的 SaaS 工具。

### 0.2 V1 目标

- 帮助用户在 **城市 + 行业** 维度筛选一批目标商家；
- 结合商家业务描述 + 用户服务描述，**批量生成定制化冷启动邮件**；
- 通过第三方邮件服务发送，并获取基础统计数据（送达、退信、打开、点击）；
- 控制产品使用门槛，让不懂技术的本地 B 端服务商在 30 分钟内完成第一次小规模 Campaign。

### 0.3 V1 范围（In Scope）

- 内置澳大利亚商家库（城市 + 行业 + 邮箱 + 业务描述）。
- 商家筛选、目标列表管理。
- 用户服务配置（服务简介、卖点、默认邮件风格等）。
- AI 批量邮件生成（按商家个性化）。
- 使用第三方邮件服务（如 SendGrid / SES 等）进行批量发送。
- 邮件事件基础统计：
  - 送达 / 退信；
  - 打开（基于追踪像素，估算）；
  - 点击（基于追踪 URL）。
- 订阅套餐：按 **月发送总量** 收费（300 / 月，1000 / 月）。
- 配额与账单周期逻辑（包括提前进入下一个计费周期）。
- 额度使用情况展示（如 `450 / 1000 封`）。
- Sender 策略：默认公共域名 + 进阶自有发件域名配置。
- Unsubscribe / suppression list：
  - 每个账号自己的退订 / 屏蔽列表（租户级），发送前过滤。

### 0.4 非 V1 范围（Out of Scope / 未来版本）

- 基于行为的智能跟进（自动 Follow-up）：
  - 按“已打开未回复”“已点击未回复”等行为自动分组、自动生成跟进邮件。
- 导入用户自有客户列表，并自动匹配/补全本地商家信息。
- 长期 nurture / re-engagement 邮件（长期培育潜在客户）。
- 后台手工赠送额度能力（运营 / 客服工具）。
- 安全 / 隐私 / 合规的完整细化（需单独文档与法律顾问确认）。
- 高级自动化流程（多步 workflow、A/B 测试等）。

## 1. 产品背景与目标

### 1.1 产品背景

在澳大利亚做 ToB 生意的本地服务商和 B 端团队，普遍缺乏有效的邮件推广手段。

典型问题：

- **无法进行有效 ToB 推广**
  很多团队只有零散的邮箱列表，或完全没有系统化的邮件营销方式，很难以结构化、可持续的方式触达到足够多的目标商家。
- **内容不够贴合目标商家**
  为了省时间，常用统一模板群发，邮件内容与收件商家的真实业务场景关联度不高，看起来像“泛泛而谈的广告”，容易被忽视或删除。
- **时间与人力成本高**
  如果想认真“做深一批客户”，往往需要：
  
  1. 逐个查商家信息（行业、店铺类型、位置、服务内容）；
  2. 再去手写匹配对方场景的邮件内容。
     这对中小团队来说非常耗时，导致“明知道邮件是条路，却没精力做好”。

### 1.2 解决方案概述

本产品是一款面向澳大利亚 B 端市场的 **AI 邮件营销工具**，主要服务对象为：

- 本地线下 ToB 服务商（如清洁、装修、维修、培训、IT 支持、营销服务等）；
- 面向澳大利亚商家提供产品或服务的线上团队 / SaaS 公司 / 代理机构。

**核心能力：**

1. **内置商家库**
   
   - 预置 > 10,000 个澳大利亚商家数据，包括：
     - 名称、城市、地址、邮箱；
     - 行业 / 业务类别；
     - 业务描述（文本）。
   - 用户可按城市、行业筛选目标商家，快速形成一批“目标名单”。
2. **AI 定制化邮件生成**
   
   - 用户提供：
     - 自己的服务简介；
     - 目标客户类型、价值主张；
     - 邮件语气、CTA 类型等。
   - 系统将：
     - 结合目标商家的业务描述、行业、城市；
     - 调用外部大模型 API，为每个目标商家生成一封 **个性化开发邮件**（标题+正文+CTA）。
3. **批量发送与效果统计**
   
   - 使用第三方邮件服务（如 SendGrid / SES）发送邮件；
   - 记录每封邮件的：
     - 送达 / 退信；
     - 打开（追踪像素）；
     - 点击（追踪 URL）。
4. **退订与 suppression list**
   
   - 每封邮件提供退订链接；
   - 收件人退订后，对应发件账号不再向该邮箱发送营销邮件；
   - 使用租户级 suppression list 在发送前过滤退订/无效邮箱。

> 注意：退订仅针对 **当前发件账号** 生效，不会从共享商家库中删除该商家，也不影响其他用户向该商家发送邮件。

### 1.3 目标用户

**（1）核心目标用户**

- 在澳大利亚向其他商家提供服务或产品的团队，包括：
  - 本地线下 ToB 服务商：
    - 清洁、维修、装修、安保、物业、培训、咨询、IT 运维、营销服务等；
  - 面向澳洲商家的线上团队 / SaaS 产品 / 代理公司：
    - POS / 预约 / CRM 等 SaaS；
    - 广告 / 数字营销代理；
    - 电商服务商等。

**（2）共同特征**

- 主要客户是 **澳大利亚境内的商家 / 门店 / 公司**；
- 希望通过冷启动邮件开发更多 B 端客户，但存在：
  - 找客户邮箱过程琐碎、低效；
  - 不擅长写高质量英文开发邮件，或认为太耗时；
  - 缺乏系统化工具查看送达、打开、点击、回复等效果，更谈不上做精细跟进。

### 1.4 成功指标（初版）

**核心目标**：提升用户进行 ToB 邮件推广的有效性，体现在“更能送达、更愿意打开、更愿意点击/回复”。

1. **送达 / 拒收**
   
   - 指标方向：
     - 在合理控制发送频率、采用专业发送服务的前提下，将退信率控制在行业合理水平，并优于用户原有方式。
   - 关注：
     - 有效邮箱占比；
     - 是否出现异常退信/被封锁风险。
2. **打开率（Open Rate）**
   
   - 目标方向：
     - 相比用户自己此前的冷邮件打开率，显著提升（具体基线后续补充）。
   - 示例：
     - 打开率目标：≥ 40%（按行业可调整）。
3. **点击率 / 回复率（Click & Reply Rate）**
   
   - 点击率：
     - 衡量邮件内容与 CTA 的吸引力；
   - 回复率：
     - 衡量真实互动比例；
   - 示例：
     - 典型用户回复率提升至 3–5% 区间（按行业细分目标）。
4. **效率**
   
   - 典型用户每小时可发送的 **包含一定个性化内容** 的开发邮件数量：
     - 从人工约 5–10 封 / 小时 提升到 ≥ 25–30 封 / 小时；
   - 每封邮件平均编写时间：
     - 从 5–10 分钟/封 降低到 ~2 分钟/封 以内（主要时间用于抽查与微调）。

## 2. 用户与使用场景

### 2.1 用户细分

#### 2.1.1 线下 ToB 服务商

- 典型角色：
  - 清洁 / 维修 / 装修公司；
  - 安保、物业服务；
  - 本地培训、咨询机构；
  - IT 运维 / 设备维护服务等。
- 核心诉求：
  - 在所在城市内快速找到匹配的目标商家；
  - 用“看起来像专门写给我”的邮件打开对话，而不是模板群发；
  - 想知道：邮件有没有送达、有没有被打开、有没有人点击、有没有人回复。

#### 2.1.2 面向澳洲 B 端市场的线上团队 / SaaS / 代理公司

- 典型角色：
  - 面向本地商家的 SaaS 公司（POS、预约系统、CRM 等）；
  - 数字营销 / 设计 / 广告服务团队；
  - 代运营、营销代理等。
- 核心诉求：
  - 稳定、可复用的 B2B 冷邮件获客方式；
  - 按城市、行业等维度做细分触达；
  - 低门槛跑 Campaign，而不必搭一整套复杂营销自动化系统。

### 2.2 V1 核心场景

> V1 聚焦：帮助用户在某个城市内，快速完成第一轮高质量冷启动开发邮件发送。

#### 2.2.1 场景一：从 0 到第一轮冷启动开发邮件

**场景说明**

- 用户：在 Melbourne 做本地清洁服务的小公司老板；
- 目标：向 Melbourne 城市内的写字楼、零售店、健身房发起一轮开发邮件，推广定期清洁服务。

**触发条件**

- 用户首次注册 / 首次使用产品；
- 或用户针对新的城市+行业组合发起新 Campaign。

**用户流程（概要）**

1. 配置“我的服务”：
   
   - 公司 / 品牌名称；
   - 服务内容与优势；
   - 覆盖城市；
   - 目标客户类型（写字楼 / 零售 / 健身房等）；
   - 核心卖点、联系方式及签名。
2. 筛选目标商家：
   
   - 选择城市：Melbourne；
   - 选择行业/类型：写字楼、零售店、健身房等；
   - 系统返回匹配的商家列表（名称、城市、邮箱、行业/描述）。
3. 选定发送对象：
   
   - 批量勾选 / 排除某些商家；
   - 可保存为「目标列表」。
4. AI 生成个性化邮件：
   
   - 选择邮件语气、CTA 类型；
   - 系统对每个商家生成个性化内容：
     - 标题；
     - 正文（包含与对方业务相关的场景 & 痛点）；
     - 明确 CTA；
     - 签名。
5. 抽查与微调：
   
   - 抽查样本邮件；
   - 对个别邮件进行编辑或重新生成；
   - 统一签名、退订文案等。
6. 发送与排程：
   
   - 设置发送时间（立即 / 定时）；
   - 系统通过第三方邮件服务发送邮件；
   - 记录外部 Message ID 用于后续事件追踪。
7. 结果记录与展示：
   
   - 发送总数、送达数、退信数及比例；
   - 打开数与打开率；
   - 点击数与点击率；
   - （未来版本可加：回复数与回复率）。

### 2.3 未来场景（V2+）

- 场景二：基于行为的智能跟进（Follow-up）；
- 场景三：导入自有客户列表、匹配补全信息；
- 场景四：长期 nurture / re-engagement 邮件。

## 3. 核心功能需求（V1）

### 3.0 范围说明

本章仅描述 V1 必须实现的功能，用于支撑：

- 城市 + 行业内筛选目标商家；
- 批量生成个性化冷启动邮件；
- 使用第三方邮件服务完成发送；
- 获取基础统计数据：送达、退信、打开、点击；
- 实现订阅套餐与配额管理；
- 实现退订 / suppression list 过滤。

---

### 3.1 商家数据与目标筛选

#### 3.1.1 内置商家库

**用户视角**

- 在后台可以看到简要介绍（例如覆盖城市数、商家数）；
- 无法直接修改内置商家数据，但可以基于筛选结果创建自己的“目标列表”。

**系统字段（最小集）**

- 商家 ID（内部唯一标识）；
- 名称；
- 城市（City）；
- 地址（可选）；
- 邮箱；
- 行业 / 业务类别（预定义枚举）；
- 业务描述（文本，供 AI 使用）。

#### 3.1.2 商家筛选与目标列表

**用户视角**

- 在“商家选择”界面可：
  - 选择城市（单选 / 多选）；
  - 选择行业 / 类别（多选）；
  - 点击“查看结果”得到商家列表；
  - 勾选 / 取消勾选目标商家；
  - 保存为“目标列表”（可命名为 “Melbourne-健身房”等）。

**系统要求**

- 支持基于城市 + 行业的快速查询；
- 目标列表字段：
  - 列表 ID；
  - 列表名称；
  - 创建时间、创建人；
  - 商家 ID 集合；
- 可设置每账号目标列表数量上限（例如 50 个）。

---

### 3.2 用户服务配置（“我的服务”）

#### 3.2.1 服务资料配置

**用户视角**

在“我的服务”页面填写：

- 公司 / 品牌名称；
- 主要服务内容（文本，有字数建议）；
- 覆盖城市（多选）；
- 目标客户类型（写字楼 / 零售 / 健身房等）；
- 核心卖点（2–3 条）；
- 联系方式 & 签名信息（姓名、职位、电话、网站等）。

**系统要求**

- 各字段结构化存储，作为 AI 生成输入；
- 更新“我的服务”不会 retroactively 改历史 Campaign，仅影响新建 Campaign。

#### 3.2.2 邮件风格与默认设置

**用户视角**

可设置默认邮件风格：

- 邮件语气：正式 / 友好 / 轻松；
- 默认 CTA 类型：
  - 邀请回复；
  - 预约电话 / Meeting；
  - 点击链接填写表单（用户提供 URL）。

**系统要求**

- 默认配置在创建 Campaign 时自动带入；
- 在具体 Campaign 中可以覆盖。

---

### 3.3 Campaign 创建与 AI 邮件生成

#### 3.3.1 创建 Campaign

**用户视角**

- 在“Campaign 管理”页点击“新建 Campaign”；
- 填写：
  - Campaign 名称（必填）；
  - 简短描述（可选）；
  - 选择目标城市、行业；
  - 选择目标商家来源：
    - 实时筛选；
    - 选择已有目标列表。

**系统要求**

- Campaign 属性：
  - Campaign ID；
  - 名称、描述；
  - 关联城市、行业；
  - 快照式目标商家 ID 集合；
  - 所属用户；
  - 状态（草稿 / 已发送 / 发送中 / 失败）。

#### 3.3.2 AI 批量生成个性化邮件

**用户视角**

- 在 Campaign 中设置生成参数：
  - 是否允许不同商家使用不同标题；
  - 邮件语气（可覆盖默认）；
  - CTA 类型；
  - 本轮想强调的促销/活动（可选）。
- 点击“生成邮件内容”。
- 看到生成进度：如“已生成 23 / 50 封”。
- 可查看每个商家对应邮件：
  - 标题；
  - 正文；
  - 支持单封编辑、单封重新生成。

**系统要求**

- 按批次（如每批 50 封）并发调用 LLM API；
- 输入包括：
  - 商家业务描述、行业、城市；
  - 用户服务描述、卖点；
  - 邮件语气、CTA 设置；
- 存储每封邮件：
  - 商家 ID；
  - 标题；
  - 正文；
  - 用户是否编辑过；
- 失败记录要标记，并支持单条重试。

#### 3.3.3 批量生成性能要求

- 对收件人数 ≤ 50 的 Campaign：
  - 95% 情况下，批量生成 ≤ 2 分钟；
- 对收件人数 ≤ 300 的 Campaign：
  - 95% 情况下，批量生成 ≤ 5 分钟；
- 对收件人数 ≤ 1000 的 Campaign：
  - 95% 情况下，批量生成 ≤ 10 分钟；
- 前端必须实时显示生成进度；
- 单个商家失败不影响整体完成。

### 3.4 发送人身份与发件邮箱配置（Sender 策略）

#### 3.4.1 默认发件方式（公共域名）

**用户视角**

- 无需配置自有域名即可发送；
- 系统一律使用平台公共域名发送，例如：
  - From：`"{{用户公司}} via {{产品名}}" <no-reply@mail.product.com>`
  - Reply-To：用户的业务邮箱（来自“我的服务”配置）。

**系统要求**

- 公共域名在邮件服务商处完成 SPF/DKIM 等认证；
- 所有新账号默认绑定公共域名。

#### 3.4.2 自有发件域名配置（高级）

**用户视角**

- 在“账号设置 / 发件设置”中可选择“使用自己的发件域名”；
- 输入 `yourcompany.com`；
- 根据向导添加 DNS 记录（TXT / CNAME 等）；
- 验证通过后可设置 `sales@yourcompany.com` 等发件地址；
- 在 Campaign 中选择自有域名发件。

**系统要求**

- 与 ESP 的域名验证 / 认证 API 对接；
- 存储租户自有域名及其验证状态；
- 发件时根据选择使用对应域名配置。

#### 3.4.3 与用量的联动（策略）

- 默认使用公共域名即可发首批 Campaign；
- 当累计发送量达到阈值（如 ≥ 300 封）或升级到付费套餐时，弹出提示推荐配置自有域名；
- 后续可对公共域名发送量 / 速率施加限制，对已认证自有域名放宽。

---

### 3.5 邮件发送与排程

#### 3.5.1 发件配置

**用户视角**

- 配置：
  - 发件人名称（From Name）；
  - 发件邮箱（From Email：公共域 / 自有域）；
  - Reply-To 邮箱。

**系统要求**

- 与 ESP 对接，保存 Sender / API key / 验证状态；
- 检查发件邮箱属于已验证域名。

#### 3.5.2 发送策略与任务队列

**用户视角**

- 在 Campaign 中选择：
  - 发送时间：立即 / 定时；
- 点“发送”后看到状态改变为“发送中”。

**系统要求**

- 所有发送通过任务队列执行；
- 创建发送任务：
  - 每封邮件一条任务；
  - 保存外部 Message ID；
- 遵守 ESP 的限流与配额要求；
- 用户点击“发送”后 30 秒内必须完成任务入队并开始提交。

### 3.6 账号、配额与套餐

#### 3.6.1 账号体系

- 用户通过邮箱注册 / 登录；
- 每个账号拥有：
  - 自己的服务配置；
  - 自己的 Campaign；
  - 自己的目标列表；
  - 自己的 suppression list（退订、退信等）。

#### 3.6.2 套餐与配额（按“月发送总量”）

**收费模型**

- 订阅制 + 按月发送量分级；
- 统计口径：当月通过系统提交到 ESP 的邮件数量（不区分送达/退信）。

**V1 套餐示例**

1. Starter
   - 300 封邮件 / 月；
   - $79 / 月。
2. Growth
   - 1,000 封邮件 / 月；
   - $99 / 月。

#### 3.6.3 额度使用与超限策略

**额度统计**

- 以当前计费周期内 **已发送封数** 为准；
- 不区分送达/退信。

**额度检查与超限规则**

- 若 `已发送数量 ≥ 套餐上限` → 当前周期视为额度已用完；
- 用户发起发送时：
  - 若额度未用完：正常发送；
  - 若额度已用完：
    - 提示本周期额度已用完，展示：已发送 / 上限；
    - 提供两种选择：
      1. 取消发送；
      2. 提前进入下一个付费周期：
         - 立即收取下一周期费用；
         - 立即重置发送额度（如 1000 封）；
         - 计费周期起止日期从“今天”重新算，后续账单日也更新。

**二次确认**

- 用户选择“提前进入下一个付费周期”时，必须有确认弹窗说明：
  - 当前周期提前结束；
  - 将立即产生下一周期费用；
  - 新周期起止日期；
  - 新额度上限。

#### 3.6.4 额度展示（如 450 / 1000）

**展示位置**

- 顶部/侧边栏简要提示：
  - `本周期：450 / 1000 封`；
- “账号 / 订阅”页面展示：
  - 套餐名称；
  - 计费周期起止；
  - 本周期使用情况：`450 / 1000 封` + 进度条。

**显示规则**

- 每次发送任务提交后更新；
- 升级/切换套餐或提前进入下一周期时重置或调整；
- 当 `已发送 ≥ 上限` 时以醒目颜色显示“本周期额度已用完”。

### 3.7 退订（Unsubscribe）与 suppression list

#### 3.7.0 作用范围原则

- 退订仅针对 **当前发件账号** 生效；
- 不删除该商家在共享商家库中的记录；
- 不影响其他账号向同一邮箱发送自己合法的邮件。

#### 3.7.1 邮件中的退订方式

- 所有商业邮件底部必须包含退订链接：
  - 文案示例：
    `If you don’t want to receive further emails from {{发件人公司}}, you can unsubscribe here: <link>`
- 点击打开退订页面，用户一键确认。
- 退订写入该账号的 suppression list，立即生效。

#### 3.7.2 退订流程

- 链接中包含：
  - tenant_id（发件账号）；
  - email（或可反解 email 的 token）；
  - option：campaign_id。
- 用户确认后：
  - 在 `(tenant_id, email)` 维度 upsert type = UNSUBSCRIBED；
  - 页面显示退订成功。

#### 3.7.3 suppression list 模型

- 每个账号有自己的 suppression list：
  - tenant_id；
  - email；
  - type：
    - UNSUBSCRIBED；
    - BOUNCED（硬退信 / 多次软退信）；
    - COMPLAINED（未来可能）；
  - source：LINK_CLICK / BOUNCE_EVENT / ADMIN_IMPORT 等；
  - created_at / updated_at；
  - note（可选）。

#### 3.7.4 发送前过滤

- 实际发送集合 = 目标商家邮箱集合 - 当前 tenant 的 suppression list 邮箱集合；
- 不查询其他 tenant 的 suppression list；
- 被过滤掉的地址：
  - 不调用 ESP API；
  - 在当前 Campaign 报表中计为“已屏蔽”。

#### 3.7.5 后台管理页面

- 提供“退订与屏蔽列表”页面：
  - 列表字段：
    - email；
    - type；
    - source；
    - created_at；
  - 支持搜索和 CSV 导出。
- 支持：
  - 手工添加某 email 至 suppression list；
  - 手工移除（仅在有明确再同意的前提下，提供备注 + 二次确认）。

## 5. 非功能性需求（节选）

### 5.1 性能

#### 5.1.1 前端交互响应

- 95% 的普通页面请求（不含 AI 生成和大报表）服务端响应时间 ≤ 2s；
- 99% ≤ 5s；
- 5xx 错误率 < 0.1%。

#### 5.1.2 AI 邮件生成

- 单封重生成：
  - 95% 请求完整生成 ≤ 8s，>10s 要有超时提示与重试入口；
- 批量生成：
  - ≤ 50 封：95% 情况下 ≤ 2 分钟；
  - ≤ 300 封：95% 情况下 ≤ 5 分钟；
  - ≤ 1000 封：95% 情况下 ≤ 10 分钟；
- 前端必须展示生成进度（已生成 X / 总数）。

#### 5.1.3 发送吞吐与稳定性

- 用户点击“发送”后 30 秒内完成任务入队并开始提交；
- ≤ 1000 封的 Campaign，应在 10 分钟内将邮件全部提交给 ESP（不含 ESP 内部延迟）；
- 遵守 ESP 的限流规则，出现限流/错误需自动重试，并在 UI 显示部分失败原因。

#### 5.1.4 报表更新延迟

- 95% 的事件在 1 分钟内入库；
- 95% 的报表更新延迟 ≤ 5 分钟（从事件发生到用户报表可见）。

### 5.2 可用性（与 sender 相关）

- 新用户在无培训情况下，应能在 15–30 分钟内完成一次 20–50 封的 Campaign；
- 首次使用不强制配置自有域名 sender，默认使用公共域名；
- 累计发送量达到一定阈值或升级后，引导用户配置自有域名，并提供向导式 DNS 说明；
- 所有长耗时操作（AI 生成、发送、报表刷新）必须有清晰的状态和进度提示；
- 高风险操作（删除 Campaign、提前进入下个计费周期等）必须有包含后果说明的二次确认弹窗。

### 5.3 可靠性（高层）

- 核心 API 月度可用性 ≥ 99.5%；
- 所有批量任务（生成/发送）通过任务队列执行，任务有明确状态，不允许“静默失败”；
- 对 AI 生成和发送任务实现合理重试与幂等，避免重复发送/重复计数；
- 邮件事件处理采用“至少一次”语义，报表端聚合时去重；
- suppression list、发送记录、事件数据保存周期至少 12 个月。
